SECTION 01
Executive Summary
Strategic overview of the Payment Switching Engine and its real-world relevance.
What This System Does
The Payment Switching Engine v3.0 is a production-grade, multi-channel payment routing and processing
platform. Built entirely in C++20 and deployed as an HTTP API server, it functions as the central nervous
system of a banking infrastructure — receiving incoming payment transactions, routing them to the correct
processing channel, validating card and account credentials, applying fraud detection rules, and returning
authorisation decisions in real time.
The system exposes a single unified API endpoint (POST /transaction/initiate) that accepts JSON payloads
and dispatches them across seven payment channels: ATM, POS terminal, E-Commerce, Mobile Banking,
QR Code payments, RingPay (wearable/contactless), and Card Issuance. This design mirrors the
architecture of real-world payment switches deployed by financial institutions and third-party processors
globally.
Problem It Solves
■ Multi-channel complexity: Modern banks must support a dozen different payment interfaces. Without
a unified switch, each channel would require its own standalone server, leading to duplicated logic,
inconsistent fraud controls, and higher operational cost.
■ Concurrency at scale: Payment volumes at peak hours can spike to thousands of transactions per
second. A synchronous, single-threaded system would immediately become a bottleneck. This engine
uses a 1,000-worker thread pool with a 30-connection database pool to handle massively concurrent
workloads.
■ Security and compliance: PAN (card number) data must never travel unencrypted. The engine
enforces AES-256-GCM encryption on every card transaction and derives PINs cryptographically without
ever storing them in a database.
■ Fraud prevention: The integrated Falcon engine runs four layered fraud rules on every transaction
before any money movement occurs, providing real-time protection comparable to dedicated fraud
platforms.
Real-World Relevance

3.2 Payment Routing Engine
The routing engine is responsible for the single most critical decision in payment processing: which channel
should handle this transaction? The design is deliberately simple yet highly extensible. The channelId in the
request payload acts as the routing key.
■ The routing key (channelId) is extracted from the outer JSON wrapper before the data payload is
passed to any processor.
■ Validation occurs at the routing layer — unknown channels fail fast with a 400 error before any DB
connection is acquired.
■ Each channel handler receives only the inner data object, maintaining clear separation of concerns.
■ The 5-second std::async timeout wrapper ensures no single slow transaction can monopolise a thread
indefinitely.

ECOM Channel — Domestic vs International Scope
The ECOM processor adds a transaction scope classification (DOMESTIC or INTERNATIONAL) by
comparing the request currency with the account's registered currency. This metadata is stored in the
transaction record and can be used downstream for regulatory reporting or currency conversion workflows.
The ECOM channel also supports a direct account number flow (without card data) for backward
compatibility with merchant integrations that manage their own tokenisation. When a card object is present,
the full decrypt → verify → lookup path is executed.
RingPay Channel — Token-Based Wearable Payments
RingPay processes transactions from wearable devices (smart rings, NFC bands) using a pre-registered
token rather than a raw PAN. The token is resolved to an account via the ringpay_tokens table, which also
enforces per-device spending limits (daily limit: ■5,000; single transaction limit: ■2,000). This design isolates
wearable payment credentials from the core card infrastructure, reducing PCI scope.

3.5 Account Lock Manager (AccountLockManager.h)
The AccountLockManager prevents race conditions when multiple concurrent transactions attempt to debit or
credit the same account simultaneously. It uses a priority queue per account to ensure CREDIT operations
(deposits, refunds) are always processed before DEBIT operations (purchases, withdrawals) when both are
waiting.
Priority
Transaction Types
Behaviour
CREDIT (0 — highest)
Deposit, Refund, Fund-in
Proceeds immediately if account is unlocked
DEBIT (1 — lower)
Purchase, Withdrawal, Fund-out
50ms initial delay; waits if any CREDIT is active
The lock is held only for the duration of the critical section (balance check → DB update → commit). It is
automatically released when the ScopedLock destructor fires, providing deadlock-free, fair, priority-aware
concurrent access per account.

The ATM channel handles WITHDRAWAL (debit) and DEPOSIT (credit) transactions. The ATM hardware
sends card and PIN data alongside the ATM/terminal identifiers. A deposit uses TxnPriority::CREDIT
ensuring it takes precedence over simultaneous debit requests on the same account.
■ Step 1: Client sends encrypted PAN + PIN + ATM/terminal IDs + amount + fee
■ Step 2: PAN decrypted, PIN verified via HMAC derivation (same mechanism as POS)
■ Step 3: Card expiry validated against database record
■ Step 4: Falcon fraud check runs with FalconChannel::ATM
■ Step 5: AccountLockManager DEBIT lock acquired after 50ms delay
■ Step 6: Balance check: must have balance >= (amount + fee)
■ Step 7: Account debited by (amount + fee); transaction_atm and transactions tables updated in single
DB transaction
■ Step 8: ATM machine receives SUCCESS response and dispenses cash
4.4 Mobile Banking — Fund Transfer Between Accounts
Mobile banking fund transfers use the encrypted PAN as the source account identifier rather than a direct
account number. The recipient is specified via creditAccount.accountNumber. The system performs a
dual-leg update: debiting the source account and crediting the destination account within the same database
transaction to ensure atomicity.
Key difference from other channels: Mobile transfers involve two accounts. The AccountLockManager
handles this by first acquiring the debit lock (with DEBIT priority) on the source account, then the credit lock
(CREDIT priority) on the destination account. The 50ms DEBIT delay ensures that if a concurrent deposit is
in-flight on the source account, it completes first.

The server uses cpp-httplib's built-in ThreadPool mechanism configured for 1,000 worker threads. This is a

6.1 PAN Encryption — AES-256-GCM
The Primary Account Number (PAN) — the 16-digit card number — is the most sensitive data element in
payment processing. Under PCI-DSS requirements, PANs must never be transmitted or stored in plaintext.
The engine enforces this through AES-256-GCM Authenticated Encryption applied at the source (client
application or test tool) before the PAN enters the network.

6.2 PIN Derivation Engine — Zero-Storage PIN
The PIN system is architecturally significant: PINs are never stored anywhere in the database. Instead,
each PIN is deterministically derived from the cardholder's PAN using a cryptographic key derivation function.
Verification re-derives the PIN and compares it with the input.
Step
Operation
Detail
1
HMAC-SHA256
Compute HMAC(PAN, secretKey) → 256-bit hash
2
Compress
Reduce hash to 64-bit value using bit rotation (rotl) and avalanche mixing
3
Multi-round derive
6 rounds of HMAC computation, each feeding previous state into next
4
State mixing
XOR all 6 derived blocks through avalanche function
5
Character selection
Index into pool (A-Z, a-z, 0-9, specials) using rotated state
6
Complexity enforcement
Positions 0-3 forced to uppercase, lowercase, digit, special
7
Verification
Re-run generation from PAN; compare with submitted PIN
Security Advantages of Zero-Storage PIN
■ No database exposure risk: A complete database breach cannot expose any PIN since they are
never written to storage.
■ Deterministic verification: The same PAN always derives the same PIN with the same secret key,
enabling reliable re-verification.
■ High entropy: 6 rounds of HMAC with avalanche mixing produce a PIN with high computational cost
for brute-force attacks.
■ Secret key protection: The security of the entire PIN system depends entirely on the secret key
(currently ULTRA_SECRET_KEY_2026_!@#) being kept confidential.

SECTION 07
Falcon Fraud Detection Engine
Real-time, multi-rule fraud prevention covering all seven payment channels.
The Falcon engine is the system's integrated fraud detection module, named after the real-world FICO Falcon
Fraud Manager used by major banks globally. It evaluates every money-out transaction against four
sequential rules before any account operation is performed. A single rule trigger results in immediate
transaction decline and full audit logging.
7.1 Fraud Rules — Ordered Evaluation
Rules are evaluated in order. The first rule that triggers causes an immediate DECLINE. Subsequent rules
are not evaluated, optimising performance and ensuring predictable behaviour.

7.2 Rule Implementation Details
Rule 1: Same-Second Duplicate Detection
Uses a per-table SQL query checking for any transaction record with the same account_number where
created_at >= NOW() - INTERVAL 1 SECOND. This catches accidental duplicate submissions from network
retries, double-clicks, and replay attacks.
SQL — SAME-SECOND DETECTION
SELECT 1 FROM transaction_pos
WHERE account_number = ?
AND created_at >= NOW() - INTERVAL 1 SECOND
LIMIT 1
Rule 2: Per-Channel Velocity
Counts transactions on a specific channel within the last 60 seconds. Detects card-testing attacks where
fraudsters rapidly run small transactions on a single channel to verify a compromised card before escalating
to larger amounts.

Rule 4: Amount Spike Detection
Computes the average transaction amount across all channels over the past 7 days. If the current transaction
exceeds 3× this average (the FALCON_SPIKE_MULTIPLIER constant), it is flagged as an anomaly. This
catches account takeover scenarios where a fraudster immediately attempts a large withdrawal after
compromising credentials.
7.3 Fraud Logging & Audit Trail
Every detected fraud event is written to the transaction_falcon table and the central transactions master
table with status=DECLINED. The log record captures:
■ transaction_id and client_txn_id — for correlation with the originating request
■ device_id and mobile_number — channel-specific metadata for pattern analysis
■ account_number and amount — for trend monitoring and alert thresholds
■ fraud_reason — human-readable rule description (e.g., "Amount spike detected: 15000 exceeds 3×
7-day average")

SECTION 11
System Strengths & Future Recommendations
Assessment of current capabilities and a roadmap for production hardening.
11.1 System Strengths
✓ Unified multi-channel API: Seven payment channels behind a single endpoint, dramatically reducing
integration complexity for client applications and providing a single point of monitoring.
✓ Zero-storage PIN system: An innovative security design that eliminates an entire class of database
breach risk. No PIN can be extracted from a database compromise because no PIN exists in the
database.
✓ Enterprise-grade concurrency: 1,000-thread pool with a 30-connection self-healing database pool
provides strong burst handling while the RAII patterns ensure clean resource management.
✓ Layered fraud protection: Four complementary fraud rules at increasing granularity (duplicate,
per-channel, cross-channel, amount spike) provide defence-in-depth without requiring an external fraud
service.
✓ Operational observability: Built-in Prometheus-compatible /metrics endpoint and structured
per-request logging with request IDs enables immediate integration with monitoring stacks like
Grafana/Prometheus.
✓ Atomic transaction integrity: Every money movement is wrapped in a MySQL transaction with
rollback on failure, ensuring no partial state is persisted even during system crashes.
✓ Open/Closed architecture: Adding a new payment channel requires zero modification of existing code
— only an addition to the dispatch map and a new processor module.
